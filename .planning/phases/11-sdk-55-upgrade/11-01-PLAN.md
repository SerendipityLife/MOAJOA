# Plan 11-01: Hermes 격리 검증 (on SDK 54)

**Phase:** 11 (SDK 54 → 55)
**Wave:** 1 (선행 — SDK bump 전 엔진 변수 격리)
**autonomous:** true (빌드타임 검증) · 런타임 로그인 육안 확인은 best-effort
**Requirements:** UPGRADE-02

## 목표

SDK 54를 **그대로 둔 채** JS 엔진만 JSC → Hermes로 되돌려, supabase-js의 OTEL 동적 import가 Hermes에서 빌드/실행되는지 **단독 검증**한다. SDK bump(11-02)와 변수를 분리해, 회귀가 나면 원인이 엔진 하나로 좁혀진다.

## 배경

`app.config.ts`가 JSC를 쓰는 유일한 이유 = supabase-js의 `/* webpackIgnore */` 등 magic comment가 붙은 동적 OTEL_PKG import가 Hermes 바이트코드 컴파일러에서 깨졌던 것(주석 기록, supabase-js@2.106.1 기준). 현재 설치본은 2.45.4이고 목표 SDK는 Hermes v1 기본 → 해소됐을 가능성이 높지만 **빌드로만 확정 가능**.

## Tasks

- **T1** — `app.config.ts`: `ios.jsEngine: 'jsc'` 라인 제거(= SDK 기본 Hermes). 위의 P0 hotfix 주석을 "Hermes 복귀(v1.2 Phase 11) — RN 0.78+ Hermes 개선으로 OTEL 이슈 해소 가정, 회귀 시 babel transform" 으로 갱신(why 기록).
- **T2** — `ios/Podfile.properties.json`: `"expo.jsEngine": "jsc"` 키 제거.
- **T3** — `pnpm --filter @moajoa/ios exec expo prebuild --clean -p ios` 로 ios/ 재생성(Hermes 반영).
- **T4** — `pnpm --filter @moajoa/ios sim` 빌드. **hermesc가 OTEL magic comment로 실패하면 fallback:** `babel.config.js`에 magic comment 제거 transform(예: `babel-plugin-transform-remove-...` 또는 커스텀 plugin) 추가 후 재빌드. 성공할 때까지 루프.
- **T5** — `xcrun simctl io booted screenshot` 으로 앱이 splash 넘어 첫 화면까지 렌더되는지 확인(supabase client import 크래시 = 화이트스크린/redbox로 드러남). 가능하면 idb로 로그인 화면 진입.
- **T6** — `pnpm --filter @moajoa/ios test` + `tsc --noEmit` 통과 확인.

## Success Criteria (verify)

1. Hermes 빌드(pnpm sim)가 hermesc 에러 없이 통과한다 — OTEL 이슈 미재현(또는 babel transform으로 해소)
2. 앱이 시뮬레이터에서 splash 넘어 첫 화면까지 렌더된다 (스크린샷 증거)
3. jest + tsc 통과 (회귀 0)
