# MOAJOA — 제품 이해 정리

> 이 문서는 코드베이스·`ARCHITECTURE.md`·`WORKSTREAMS.md`·`constants.ts`를 바탕으로
> "지금 MOAJOA가 무엇을 지향하는 서비스인가"를 한 장에 정리한 것이다.
> 제품 합의 문서가 아니라 **현재 코드/문서에서 읽히는 의도**의 요약이다. 틀린 게 있으면 PR로 교정.

---

## 1. 한 줄 정의

**MOAJOA는 "영상·글 속에 흩어진 장소를, 던지기만 하면 지도 보드로 모아주고, 친구와 같이 정하게 해주는" 도구다.**

> 유튜브·블로그·인스타 링크를 공유하면 → 영상/글 속 가게·명소를 자동 추출 → 지도 핀으로 보드화 → 친구를 초대해 ❤️ 투표로 "갈 곳"을 같이 확정.

---

## 2. 풀려는 문제 (Why)

여행/맛집 계획에서 반복되는 진짜 불편:

1. **수집의 마찰** — "이 영상에 나온 그 라멘집 어디였지?"를 일일이 댓글·자막에서 찾아 캡처·메모·지도앱에 옮겨 담는 수작업.
2. **흩어짐** — 정보가 유튜브·인스타·블로그·카톡·메모앱에 파편화되어 한 곳에 안 모인다.
3. **합의의 어려움** — 같이 가는 친구들과 "여기 갈까 말까"를 정하는 과정이 카톡 스크롤 지옥. 누가 뭘 좋아하는지 한눈에 안 보인다.

MOAJOA의 베팅: **(1) 추출을 AI가 자동화하고, (2) 지도 위 하나의 보드로 모으고, (3) 투표로 합의를 구조화**하면 이 세 마찰이 사라진다.

---

## 3. 타겟 (Who)

- **1차:** 한일 여행을 계획하는 한국어 사용자. 기본 로케일 `ko`, 보조 `ja`.
  - 지원 도시 코드에 도쿄·오사카·교토·후쿠오카·삿포로·오키나와(일본) + 서울·부산·제주(한국)가 먼저 들어있는 것에서 드러남 (`CITY_KO_MAP`).
  - 장소 이름을 `name_local`(현지어, 도쿄면 일본어)·`name_ko`(모국어)·`name_en` 3중으로 보존 → "현지에서 통하는 이름"과 "내가 아는 이름"을 동시에.
- **사용 맥락:** 유튜브 여행 브이로그/맛집 영상을 즐겨 보고, 친구·연인·가족과 함께 일정을 짜는 사람.
- **협업 단위:** 보드당 멤버 최대 20명 — 소규모 동행 그룹 기준.

---

## 4. 핵심 가치 제안 (What)

| 가치 | 어떻게 |
|---|---|
| **던지면 끝나는 수집** | iOS 공유시트 → MOAJOA → URL이 보드에 자동 저장, AI가 장소 추출 |
| **지도로 한눈에** | 추출된 장소를 핀으로. 핀 클릭 → 그 장소가 나온 **영상 타임스탬프로 점프** |
| **같이 정하기** | 보드를 친구에게 공유 → ❤️ 투표 → "확정" 필터로 갈 곳만 추림 |
| **비로그인 열람** | 카톡으로 받은 공유 링크를 로그인 없이 즉시 열람(웹 SSR) → 가입 전환 |

추출 파이프라인(유튜브 기준): 메타데이터 → 자막(ko→ja→en 폴백) → LLM(claude)이 장소 후보 추출 → Google Places로 좌표 resolve → 지도 반영. 목표 end-to-end < 30초.

## 5. 플랫폼 역할 분담 (지향하는 경계)

이 분담이 제품 철학의 핵심이라 명확히 지킨다:

- **iOS (Expo)** = **생산자/소유자 경험.** 저장·공유 수신(Share Extension)·캡처·편집·투표. "내 보드를 만들고 채우는 곳."
- **Web (Next.js)** = **소비자/획득 경험.** 공유 링크 열람·공개 보드·투표 참여. "남이 만든 보드를 보고 합류하는 곳."
  - ⚠️ Web의 "보드 생성/링크 추가" 폼은 **정식 기능이 아니라 dev tool** (`NEXT_PUBLIC_ENABLE_DEV_TOOLS=1`로만 노출). 보드를 *만드는* 행위는 iOS 전용이라는 의도가 명확하다.
- **Supabase** = 단일 백엔드. 관계형 데이터(boards↔links↔places↔votes↔memberships) + RLS(deny-by-default) + PostGIS(지도 쿼리) + Realtime(라이브 투표/추출 진행).

> 한 줄: **iOS로 모으고, Web으로 퍼뜨린다.**

## 6. 도메인 모델이 말해주는 제품 모양

`constants.ts`의 enum/limit이 제품 의도를 그대로 드러냄:

- **소스 종류** `youtube`(자동 추출) · `blog`/`instagram`(저장 후 운영 큐 추출) · `manual`(직접 추가) → 유튜브가 1급 시민, 나머지는 단계적.
- **보드 공개 범위** `private`(나만) · `shared`(초대 멤버 협업) · `public`(링크 가진 누구나 열람).
- **멤버 역할** `owner`/`editor`/`voter` → 협업이되 권한은 구분.
- **투표** 현재 `love` ❤️ 단일 종류. **"확정" = love ≥ 1**인 장소 (임계값·종류는 진화 예정).
- **한도(무료 티어)** 사용자당 보드 20 · 보드당 링크 50 · 멤버 20 · 링크당 장소 30 → MVP는 소규모 개인/그룹 사용 가정.
- **추출 상태** `pending→processing→ready/failed/manual_review` + 진행 단계 한국어 표시("영상 정보 가져오는 중" 등) → **추출이 시간 걸린다는 걸 숨기지 않고 진행률로 신뢰를 주는** UX 지향.
- **신뢰도** confidence < 0.7 핀은 "신뢰도 낮음" 처리(흐린 마커 + 확인/거절) → AI 추출의 불확실성을 사용자에게 투명하게.

## 7. 사용자 여정 (지향하는 해피패스)

```
[발견]  유튜브에서 도쿄 맛집 영상을 본다
   ↓
[수집]  공유시트 → MOAJOA → "도쿄 여행" 보드에 저장 (iOS)
   ↓
[자동]  AI가 영상 속 가게들을 핀으로 지도에 꽂아준다 (~30초, 진행률 표시)
   ↓
[정리]  핀 클릭 → 영상 그 장면으로 점프해 확인, 필요시 편집
   ↓
[공유]  보드를 shared로 → 카톡으로 친구에게 /b/<slug> 전송
   ↓
[합의]  친구가 (웹에서 비로그인 열람 → 투표는 로그인) ❤️ 투표
   ↓
[확정]  Realtime으로 표가 실시간 집계 → "확정" 필터로 갈 곳만 남긴다
```

## 8. 제품 원칙 (코드/문서에서 읽히는 태도)

- **AI는 보조, 결정은 사람.** 추출은 자동이지만 신뢰도·확인/거절·투표로 사람이 최종 결정한다.
- **마찰 최소화가 1번 지표.** "던지면 된다"가 핵심. 수집 단계에서 손이 많이 가면 진 거다.
- **공유가 곧 성장.** 비로그인 열람 가능한 공개 보드 = acquisition 엔진. 웹 SSR/OG 카드에 투자.
- **신뢰는 투명성에서.** 추출 진행 단계 노출, 낮은 신뢰도 핀 표시 — 블랙박스로 굴리지 않는다.
- **한일 여행에 먼저 깊게.** 만능을 노리지 않고 특정 사용자/지역에서 완성도를 먼저 만든다.
