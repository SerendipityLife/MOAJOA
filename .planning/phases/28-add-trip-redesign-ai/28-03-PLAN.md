<objective>
이 phase의 **유일한 미해결 설계물 D-21**(수동 배치 Day 고정)을 계약으로 확정하고, D-09(EF day_count fallback)를 함께 잠근다. 두 변경 모두 `generate-plan` EF를 건드리므로 한 플랜에 묶었다.

**문제:** `generate-plan`은 draft 플랜을 통째로 delete 후 재insert하는 **멱등 덮어쓰기**(index.ts L237)다. 그리고 `moveToDay`는 `is_anchor:false`로 넣는다 — 즉 지금은 "수동 배치"를 식별할 마커 자체가 없고, 사용자가 손으로 Day 3에 옮긴 장소가 '일정 다시 만들기' 한 번에 날아간다. `anchor_place_ids`는 "어딘가 배치"만 보장하고 "어느 날"은 보장하지 않는다.

**채택 계약 (RESEARCH Open Q1 옵션 (a) — DB 스키마 추가 0):**
1. `moveToDay`가 `is_anchor: true`로 기록한다 → 수동 배치 = 앵커 (의미 확장: "필수 + Day 고정")
2. 재생성 시 클라이언트가 현 플랜의 `is_anchor` 항목에서 `{place_id, day_index}` 쌍을 모아 EF의 **additive 요청 필드 `pinned_placements`**로 보낸다
3. EF 프롬프트가 "place X는 반드시 day Y에" 제약을 싣는다
4. LLM이 불복해도 EF가 **사후 강제**한다(`enforcePinnedPlacements`) — 기존 `validatePlanIds`(T-18-12) 방어 idiom을 그대로 미러
5. EF가 pinned 장소를 다시 `is_anchor:true`로 기록해 **루프를 닫는다** (2회차 재생성에도 고정이 산다)

옵션 (b)(EF가 수동분 보존)는 마커 컬럼이 없어 식별 불가 → 마이그레이션 추가 필요라 비채택. 옵션 (c)(클라 사후 복원)는 레이스·깜빡임·Routes leg 불일치로 비채택.

Purpose: D-25 카피("직접 옮긴 장소는 그대로 두고 나머지만 다시 짜요")를 **진실로 만든다.** 이 계약 없이 그 카피를 배포하면 거짓말이 된다.
Output: core 요청 스키마 · api moveToDay · EF 프롬프트·강제 로직·day_count fallback.
