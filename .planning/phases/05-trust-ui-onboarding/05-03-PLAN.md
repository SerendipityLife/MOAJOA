# 05-03-PLAN — iOS Error Toast Retry + Link List Failed Row (TRUST-03)

**Wave:** 3 (sequential after 05-02 because both touch [id].tsx; depends_on 05-01 for context only)
**Owners:** iOS
**Requirements covered:** TRUST-03
**Depends on:** 05-01 (no direct constant — 본 plan은 toast + boards/[id].tsx 로컬), 05-02 ([id].tsx 공유)
**Blocks:** 05-04 ([id].tsx 공유)

---

## Goal

`apps/ios/lib/toast.tsx`에 optional `action` slot을 추가하고, `[id].tsx`의 broadcast 'error' 분기가 `[재시도]` 인라인 액션 토스트를 띄우게 한다. 동시에 링크 리스트 행의 `extraction_status` visual에 5개 상태 카피 + `failed` 행 탭 = retry를 wire한다.

UI-SPEC §"Component States" 3, 4번 + CONTEXT D-10/D-11/D-12.

## Why

- TRUST-03: 추출 실패 사유 노출 + 1탭 retry — dogfooding 동안 매 실패마다 사용자가 메인 앱 재시작 비용 0
- 자동 retry는 quota burst 가중 (D-12) — 사용자 의도 retry만
- toast retry는 직후 1번, link list retry는 dismiss 후 영구 fallback (D-10/D-11 이중 표면)

## Verification Loop

```
1. YouTube URL 분석 강제 실패 (예: 자막 없는 영상)
2. error toast 등장 — 좌측 텍스트 + 우측 [재시도] 가시
3. [재시도] 탭 → step indicator overlay 재진입 (triggerExtraction 재호출)
4. toast 8초 후 자동 dismiss (Phase 3 5초 → 8초 변경)
5. 다른 시나리오: toast dismiss 후 보드 상세 링크 리스트의 failed 행 탭 → retry 동일 동작
6. ready/processing/pending/manual_review 행 탭 → no-op
```

## Tasks

### T1 — `toast.tsx` action slot 확장

**File:** `apps/ios/lib/toast.tsx`

**ToastState 인터페이스 확장:**
```ts
interface ToastState {
  id: number;
  message: string;
  kind: ToastKind;
  durationMs: number;
  action?: { label: string; onPress: () => void };   // NEW
}
```

**showToast 시그니처 변경 (backward-compatible — action optional):**
```ts
export function showToast(
  message: string,
  kind: ToastKind = 'info',
  options?: { durationMs?: number; action?: { label: string; onPress: () => void } },
): void {
  const t: ToastState = {
    id: nextId++,
    message,
    kind,
    durationMs: options?.durationMs ?? (kind === 'error' ? 8000 : 3000),  // Phase 5: error 5s → 8s (D-10)
    action: options?.action,
  };
  for (const l of listeners) l(t);
}
```

> **Breaking surface:** Phase 3 호출자 `showToast(msg, 'error')`는 변경 없이 동작. 본 plan은 옵션 3번째 인자 형태로 변경 — 기존 `showToast(msg, kind, durationMs)` 직접 사용처 점검 필요. grep:
> ```bash
> grep -rn "showToast(" apps/ios packages | grep -v "\.test\."
> ```
> 모두 2-arg 호출이면 안전. 3-arg (durationMs) 호출자가 있으면 객체 형태로 마이그레이션.

**ToastHost render 변경:** Pressable wrapper 안에 `current.action ? row layout with action button : 기존 단일 텍스트`:

```tsx
<View className={`mx-4 px-4 py-3 rounded-xl shadow-md ${bg} ${
  current.action ? 'flex-row items-center justify-between' : ''
}`} style={current.action ? { gap: 8 } : undefined}>
  <Pressable onPress={hideToast} style={{ flex: 1 }} hitSlop={8}>
    <Text className="text-white text-sm">{current.message}</Text>
  </Pressable>
  {current.action && (
    <Pressable
      onPress={() => {
        const a = current.action!;
        hideToast();
        a.onPress();
      }}
      hitSlop={8}
      className="px-2 py-1"
    >
      <Text className="text-sm font-semibold text-white underline">
        {current.action.label}
      </Text>
    </Pressable>
  )}
</View>
```

**Verify:** Phase 3 `showToast('msg', 'error')` 콜은 단일 텍스트 토스트 그대로. 새 형태로 부르면 우측 액션 버튼 표시. 8초 자동 dismiss.

### T3 — Link list row status 표시 + failed tap retry

**File:** `apps/ios/app/boards/[id].tsx`

기존 renderItem (lines 190-199)을 확장:

```tsx
renderItem={({ item }) => {
  const status = item.extraction_status;
  const isFailed = status === 'failed';
  const statusKo =
    status === 'pending' ? '분석 대기'
    : status === 'processing' ? '분석 중...'
    : status === 'ready' ? '분석 완료'
    : status === 'failed' ? '분석 실패 — 탭하여 재시도'
    : status === 'manual_review' ? '재추출 필요'
    : status;
  const onRowPress = () => {
    if (!isFailed) return;
    setAnalyzing(item.id);
    triggerExtraction(supabase, item.id).catch((err) => {
      console.warn('[row-retry] failed:', err);
      setAnalyzing(null);
      showToast('재시도 실패', 'error');
    });
  };
  return (
    <Pressable
      onPress={onRowPress}
      disabled={!isFailed}
      className="p-3 border border-neutral-200 rounded-lg mb-2"
    >
      <Text className="text-sm font-medium" numberOfLines={1}>
        {item.title ?? item.url}
      </Text>
      <Text className="text-xs text-neutral-500 mt-1">
        {item.source_kind} · <Text className={isFailed ? 'text-danger' : 'text-neutral-500'}>{statusKo}</Text>
      </Text>
    </Pressable>
  );
}}
```

**UI-SPEC §"Component States" 4 매칭:** status text는 14/400. 본 plan은 기존 row가 `text-xs` (12px)로 메타 표시 중 — 일관성 유지 위해 UI-SPEC 사이즈는 14가 권장이지만 기존 12 유지가 surgical(§3.3). 본 plan은 12 유지 + danger 색 분기만 추가. **결정:** size는 row의 기존 `text-xs`를 유지하되, UI-SPEC 사이즈 14는 향후 row 전체 typography 일관 정리 plan(별도)에서. CONTEXT D-22의 text-xs 활성 use site에 "링크 리스트 status"는 명시 X (있는 것: 배지·step indicator 미래). 따라서 12 유지가 D-22 lock과 충돌 X.

**Verify:**
- failed 행 탭 = retry 트리거. 다른 상태 disabled (탭해도 no-op).
- failed status 단어만 danger 색.

## Risk / Pitfalls

- **showToast 시그니처 변경:** grep 결과 호출자가 `showToast(msg)` 또는 `showToast(msg, kind)` 2-arg 패턴만이면 안전. 3-arg numeric durationMs 호출자가 있으면 컴파일 에러 → 객체 형태로 마이그레이션 (단 grep으로 1차 확인).
- **retry 후 또 fail:** 동일 toast 패턴으로 재진입. 사용자가 dismiss 또는 무한 retry 가능 — D-12 의도. 무한 루프 방지는 Phase 2 Edge Function의 idempotency 가드(processing 중복 차단)가 처리.
- **`analyzing` capture:** error 시 setAnalyzing(null) 이전에 linkId capture 필요 — closure 보존.
